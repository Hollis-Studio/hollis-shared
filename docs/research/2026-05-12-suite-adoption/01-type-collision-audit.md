## Top Promotion Candidates (Workouts â†’ Shared)

Ordered by cross-app utility and urgency.

1. **`MuscleGroup` (23-value set)** â€” Workouts has the anatomically complete set. Shared has 11. Every fitness-related surface in the suite needs the full set. The shared enum must grow to the Workouts definition. File: `src/types/constants.ts:285-308` and `src/schemas/exercise.ts:103-127`.

2. **`EquipmentType` (reconciled value set)** â€” Both sides have divergent values. The canonical set must be designed with the backend in mind (Prisma enum). `cable_machine` vs `cable` is the most dangerous discrepancy. File: `src/types/constants.ts:87-103`.

3. **`ProgressionBaseline` + `BaselineEntry`** â€” The entire E1RM ratchet model is Workouts' core IP. Once the backend is Postgres, the server needs to read/write this shape. Promote to `@hollis-studio/workouts-contracts` or a `./domain/progression` subpath. File: `src/types/models.ts:513-580`.

## Highest-Risk Shape Drifts

These will break consumers or require data migrations if reconciled incorrectly.

### 1. `MuscleGroup` value-set mismatch (CRITICAL)

- **Workouts** (`src/types/constants.ts:285-308` + `src/schemas/exercise.ts:103-127`): 23 values including sub-muscle anatomy (`anterior_deltoids`, `lateral_deltoids`, `posterior_deltoids`, `lats`, `traps`, `hip_flexors`, `adductors`, `abductors`, `neck`, `obliques`, `lower_back`, `upper_back`).
- **Shared** (`domain/exercise.ts:95-107`): 11 values (the top-level groups only).
- **Risk**: Workouts Firestore documents contain values like `'traps'` and `'lats'` that would fail Zod parse against the shared 11-value schema. Any code path that validates Workouts documents against the shared `MuscleGroupSchema` will throw at runtime.
- **Resolution**: Shared must adopt the 23-value set. The Prisma migration must add the new values to the `MuscleGroup` enum **before** the shared schema is published as `v0.1.0-alpha.1`. Do not reduce Workouts' set â€” it would corrupt existing data.

### 2. `EquipmentType` non-overlapping value sets (CRITICAL)

- **Workouts** (`src/types/constants.ts:87-103`): 16 values. Notable Workouts-only values: `cable_machine`, `plate_loaded_machine`, `smith_machine`, `treadmill`, `stationary_bike`, `rowing_machine`, `elliptical`, `stairmaster`, `jump_rope`, `none`, `other`.
- **Shared** (`domain/exercise.ts:143-154`): 10 values. Notable shared-only values: `cable` (not `cable_machine`), `squat_rack`, `bench`, `pull_up_bar` (not present in Workouts at all).
- **Risk**: These two enums are **not compatible** â€” `cable` â‰  `cable_machine`. A `GymProfile.equipmentTypes` array read from Workouts Firestore contains `'cable_machine'`; the shared schema expects `'cable'`. Parse would fail. Conversely, exercises imported from the Health backend may have `'squat_rack'` which Workouts has no display name for.
- **Resolution**: Before publishing shared package, design a single canonical `EquipmentType` set that satisfies both apps and the Prisma enum. Likely outcome: keep Workouts' cardio-specific values; add shared's gym accessories (`squat_rack`, `bench`, `pull_up_bar`) to Workouts; rename `cable` â†’ `cable_machine` in shared to match Workouts (if Workouts has more users). This is a **Prisma migration + Firestore data migration** in Workouts.

### 3. `ExerciseCategory` taxonomy mismatch (HIGH)

- **Workouts**: `'weightlifting' \| 'cardio' \| 'stretching'` â€” used as top-level exercise **type** to route tracking (reps vs time vs stretch).
- **Shared**: `'COMPOUND' \| 'ISOLATION' \| 'CARDIO' \| 'MOBILITY' \| 'PLYOMETRIC'` â€” used as movement **pattern classification** within exercise library.
- **Risk**: These answer different questions. Importing and using the shared type in Workouts code without renaming will cause category-filter bugs (e.g., a `'COMPOUND'` barbell squat would not route to the reps-tracking flow).
- **Resolution**: Rename the shared concept to `ExerciseMovementCategory` or `ExerciseLibraryCategory`. Introduce a new `ExerciseTrackingCategory` (or `ExerciseDomain`) in shared that matches Workouts' 3-value taxonomy. Both can coexist on the same exercise record (tracking domain + movement classification).

### 4. `UserProfile` / `WorkoutSession` name collisions with different shapes (HIGH)

- **`UserProfile`**: Workouts is a 9-field Firebase auth + settings document. Shared is a 100+ field clinical Postgres entity. Importing the shared type accidentally would silently break type checks across all hooks that access `profile.settings` (which doesn't exist on shared `UserProfile`).
- **`WorkoutSession`**: Workouts models a completed exercise log. Shared models a scheduled workout day with sections and rest-day flags. Any shared code that handles both without namespacing will confuse the two.
- **Resolution**: Both pairs need distinct names before being consumed from the same import space. Propose: shared renames its `WorkoutSession` â†’ `WorkoutPlanDay`; Workouts' `UserProfile` becomes `WorkoutsUserProfile` post-migration.

### 5. `passwordSchema` leniency gap (MODERATE)

- **Workouts** (`src/schemas/auth.ts:41-45`): 8-char min + 1 uppercase + 1 number.
- **Shared** (`password/index.ts`): Enforces `PASSWORD_POLICY` (length, complexity, zxcvbn score â‰Ą 2, optional HaveIBeenPwned check).
- **Risk**: If the Identity Service backend enforces the shared schema rules and Workouts forms use the lenient local schema, users can create passwords that pass the Workouts form but are rejected by the server (degraded signup/change-password UX).
- **Resolution**: Workouts must adopt the shared `passwordSchema` when Firebase Auth is replaced. Begin by adding it as a peer validator behind a feature flag.
