## 2. The `schemas/index.ts` and `schemas/typecheck.ts` Story

### `src/schemas/index.ts` — current role

`src/schemas/index.ts` (lines 1–74) is a **named-export barrel** for all Workouts Zod schemas. It re-exports from:

- `./exercise` — 8 schemas/types
- `./gym` — 2 schemas
- `./user` — 3 schemas
- `./program` — 6 schemas
- `./session` — 5 schemas
- `./baseline` — 4 schemas
- `./analytics` — 1 schema
- `./metrics` — 7 schemas (new as of 2026-05-11)
- `./slidePayloads` — 1 const object
- `./deterministicSnapshot` — 1 schema + type
- `./auth` — 5 schemas

It does **not** re-export the `ai/` sub-namespace (that uses `src/schemas/ai.ts` → `ai/index.ts`).

**Post-adoption recommended shape:**

After shared schemas are adopted, `index.ts` becomes a **mixed barrel**:

1. Schemas that become thin re-exports from `@hollis-studio/contracts` (e.g., `WeightUnitSchema`) should be replaced with a direct `export { WeightUnitSchema } from '@hollis-studio/contracts/domain/units'` — or removed from the barrel if consumers are updated to import from the canonical source.
2. Schemas that remain Workouts-local (user, gym, program, session, baseline, metrics, AI) stay as-is.
3. The barrel should add a block comment clearly delineating "Adopted from @hollis-studio/contracts" vs "Workouts-local".

The barrel's value for Workouts-local schemas is high — it is the single import point cited by 40+ consumers in `@consumers` JSDoc blocks across services and hooks. It should not be dissolved.

### `src/schemas/typecheck.ts` — SM-1/SM-2 enforcement

`typecheck.ts` (lines 1–170) is a **compile-time parity file** with no runtime exports beyond `SchemaTypeChecks`. It uses a custom `AssertExtends<TSchema, TModel>` pattern (line 71) to enforce that each Zod schema's inferred type extends its paired interface in `src/types/models.ts`.

Current coverage (27 pairs asserted):

- User (2), Gym (2), Exercise (3), Program (4), Session (4), Progression Lifting (2), Progression Cardio (2), Metric Basket (7: GatedE1RM, BestQualifyingSet, WeeklyHardSets, RelativeStrength, RepRangePivot, ConfidenceBand, MetricBasketSnapshot).

**How it interacts with shared schema typecheck assertions:**

`@hollis-studio/contracts` has no equivalent typecheck file — schemas in shared derive their types directly via `z.infer<>` and the inferred type IS the contract (no separate interface to check against). This is a different pattern.

**Post-adoption impact:**

When Workouts adopts a shared schema (e.g., `WeightUnitSchema`), the `typecheck.ts` assertion for the relevant model (e.g., `UserSettings`) must still hold because the `UserSettingsSchema` (which remains local) references the shared `WeightUnitSchema`. The assertion chain is:

```
AssertExtends<z.infer<typeof UserSettingsSchema>, UserSettings>
```

If shared schema field types are identical or wider, the assertion continues to pass. The risk is if a shared schema's inferred type is **narrower** than the local interface (e.g., a shared enum with fewer values). This must be spot-checked per adopted schema.

`typecheck.ts` should **gain** a second import block `from '@hollis-studio/contracts/...'` for any Workouts type that wraps a shared schema, with a comment indicating the delegation chain.

## Appendix: Zod Version Compatibility

Both repos use `zod@4.3.6`. No version-skew risk. Shared contracts is an ESM-only package (`"type": "module"`). Workouts uses Expo/Metro bundler — ensure `@hollis-studio/contracts` is compiled to CJS-compatible output or that Metro's ESM support is enabled before attempting import. This is a build concern, not a schema concern, but it will block step 1 of adoption if not resolved first.
